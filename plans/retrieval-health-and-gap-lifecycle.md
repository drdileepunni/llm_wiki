# Retrieval Health & Gap Lifecycle — Design Plan

**Date:** 2026-05-14  
**Status:** Design / pre-implementation

---

## Background & Root Cause Analysis

### Why CDS grounding fails while chat QnA succeeds for the same query

The CDS pipeline (order generation) and the chat QnA pipeline both use the same vector store and the same wiki, but they retrieve differently:

| | Chat QnA | CDS (order generation) |
|---|---|---|
| Search query | Full original question verbatim | Short phrase generated by the LLM in Step 1 |
| Threshold before passing to LLM | None — all top-k sections go in | `_MIN_RETRIEVAL_SCORE = 0.55` fallback trigger |
| Grounding judgment | LLM synthesises from wiki + training | LLM strictly checks: is the value *explicitly* in the retrieved section? |

These different search strings produce different embedding vectors, which retrieve different sections. In the furosemide case, the CDS query ("furosemide dose acute pulmonary edema CKD") retrieved a section scoring 0.84 that did not contain the CKD-specific dose. The QnA query (the full clinical question) happened to retrieve a different section that did.

**The retrieval failure is therefore a query formulation problem, not a wiki content problem.** The answer existed in the wiki; it just wasn't surfaced to the grounding LLM.

### CDS threshold landscape (for reference)

| Constant | Value | Purpose |
|---|---|---|
| `_MIN_RETRIEVAL_SCORE` | 0.55 | Below this → retry with shorter fallback query |
| Grounding prompt hint | 0.65 | Soft instruction to be more skeptical below this |
| `_GAP_COVER_THRESHOLD` | 0.72 | Step 4 marker rewrite: hit must exceed this to use |
| `_CONFIRM_RETRIEVAL` | 0.70 | Gap placement: below → "approximate", above → "confirmed" |
| `_GAP_CONFIRM_THRESHOLD` | 0.75 | `scan_text_gaps` content verification |
| `_LOW_RETRIEVAL` | 0.60 | Gap registration: below → run pre-write coverage check |

Chat QnA has no score threshold — all retrieved sections are passed to the LLM regardless of score.

## Three Problems to Solve

### Problem 1 — Retrieval failure: wrong sections surfaced

The CDS pipeline passes up to 6 sections to the grounding LLM, but only `top_score` and `pages_hit` (page paths only, no section names or individual scores) are stored in the trace and shown in the UI. The user has no visibility into what was actually retrieved.

Additionally, writing content to a wiki page does not guarantee that the CDS retrieval query will find it. The semantic gap between query phrasing and section embedding can cause a newly filled gap to be immediately unretrievable.

### Problem 2 — No gap regression after close

Once a gap is resolved, there is no verification step. The system has no way to confirm that the original query would now succeed. A gap can be marked closed while the filled section is effectively invisible to the retrieval query that opened it.

### Problem 3 — Persistent loops / query-section mismatch

The system currently tracks `gap_opens` per page (in `page_metrics.py`) but not per `(query, section)` pair. When a gap keeps reopening, it may be because:

1. Queries consistently retrieve sections A, B, C
2. The resolver fills the answer into section D
3. Section D never appears in retrieval → the gap reopens immediately

This cycle is currently undetectable. The user cannot see whether the filled section is the one being retrieved.

---

## Solution: Tie the Gap Lifecycle to the Retrieval Query

### Core principle

**Work in embedding space, not string space.**

"Furosemide dosing" and "Dosing for Furosemide" are different strings but near-identical embedding vectors. The solution must never compare query strings — it must compare embedding vectors. This handles:

- Paraphrase equivalence (same question, different wording)
- Duplicate gap detection (two gap files for the same missing knowledge)
- Persistent loop detection (same gap resurfacing under different phrasing)

## Backend Changes

### 1. Richer gap file schema

Add the following fields to gap files at **open time**:

```yaml
# existing fields unchanged
resolution_question: "What is the recommended initial IV furosemide dose..."
placement: confirmed

# NEW
query_embedding: [0.023, -0.41, ...]          # embedding of resolution_question
searched_sections:                             # all sections retrieved, not just top
  - path: entities/furosemide.md
    section: Dosing
    score: 0.84
  - path: entities/furosemide.md
    section: Indications
    score: 0.71
  - path: concepts/diuretics.md
    section: CKD Adjustments
    score: 0.68
  - path: concepts/loop-diuretics.md
    section: Mechanism
    score: 0.61
reopen_count: 0
open_events:
  - timestamp: "2026-05-14T10:23:00Z"
    search_query: "furosemide dose acute pulmonary edema CKD"
    top_score: 0.84
```

Add the following fields at **close time** (after gap resolver writes content):

```yaml
filled_section: "Renal Dosing Adjustments"    # section where content was written
filled_page: entities/furosemide.md
retrieval_verified: false                      # did re-retrieval find the filled section?
verified_score: 0.51                           # score of filled section for original query
retrieval_mismatch: true                       # searched_sections ≠ filled_section page/heading
close_events:
  - timestamp: "2026-05-14T11:00:00Z"
    filled_section: Renal Dosing Adjustments
    verified_score: 0.51
```

### 2. Duplicate / reopen detection using embedding similarity

When `write_gap_files()` is called:

1. Embed the incoming `resolution_question`
2. Cosine-compare against all stored `query_embedding` vectors in existing open gap files
3. If similarity > ~0.92 → this is the same gap under a different phrasing
   - Increment `reopen_count` on the existing gap
   - Append to `open_events` instead of creating a new file
   - Do not create a duplicate gap file

This collapses paraphrase variants into a single tracked gap.

### 3. Post-fill retrieval verification

After the gap resolver writes content to a wiki section:

1. Re-run `search_sections(resolution_question, wiki_dir, top_k=10)`
2. Check whether the newly written section appears in top results above `_GAP_COVER_THRESHOLD` (0.72)
3. Write `retrieval_verified`, `verified_score`, and `retrieval_mismatch` into the gap file

If `retrieval_verified = false`:
- The content exists but is not findable by the original query
- The gap should not be considered closed — either the content needs to move to the section that IS being retrieved, or the retrieved section needs a cross-reference

### 4. Files to change

| File | Change |
|---|---|
| `services/ingest_pipeline.py` — `write_gap_files()` | Add embedding, embed-based dedup, `searched_sections` storage |
| `services/chat_pipeline.py` — CDS Step 2 | Pass all retrieved sections (not just top score) into the trace and gap metadata |
| `services/gap_resolver.py` | Add post-fill retrieval verification step |
| `routers/kbs.py` (or graph router) | Expose `retrieval_mismatch`, `unverified_fill` flags to the graph API |

## Resolved Gap Index — Two-Tier Retrieval

### The idea

Once a gap is filled, the resolved gap should become an **active retrieval shortcut** for future semantically similar queries — not just a diagnostic record.

Currently, query embeddings in the plan are used only for dedup (detecting if a new gap is the same as an existing open gap) and post-fill verification. Neither helps future queries find the answer. This section adds a third use: **operational retrieval**.

### Why this matters

The failure mode we are solving is that query phrasing does not always align well with section embedding, even when the section has the right content. The root cause is that vector search matches **question against passage text** — two different semantic spaces. A resolved gap sidesteps this entirely by matching **question against question**: the incoming query is compared to the embedding of a previously resolved question, which is inherently a better match.

### Architecture: two-tier retrieval in CDS Step 2

Before running `search_sections()` for each parameter, check the resolved gap index first:

```
Query Y arrives at CDS Step 2
  │
  ├─ Tier 1: resolved gap index (fast lookup)
  │   embed(Y) ──cosine──► all resolved gap query_embeddings
  │   similarity > 0.92 → directly use filled_section as context
  │   ↳ skip normal vector search for this parameter
  │
  └─ Tier 2: normal section vector search (fallback for novel queries)
      runs only if Tier 1 finds no match
```

If Tier 1 matches, the matched section is passed to the grounding LLM as context exactly as normal retrieval would. The LLM still runs and reads the current live section content — this is not an answer cache. If the wiki is updated with better content, future queries automatically benefit.

### When to use Tier 1

Use Tier 1 regardless of whether `retrieval_verified` is true or false on the matched gap:

- `retrieval_verified = true` → Tier 1 is a fast path to what normal retrieval would eventually find
- `retrieval_verified = false` → Tier 1 is the **only reliable path**: the filled section exists but normal retrieval can't find it. The shortcut routes around the retrieval failure entirely.

The similarity threshold (~0.92) should be higher than the dedup threshold (~0.92 is fine for both — the use cases are the same: semantic equivalence, not loose similarity).

### The resolved gap index

A separate JSON file (`resolved_gap_index.json` in the wiki directory) storing only entries where the gap has been filled:

```json
[
  {
    "query_embedding": [...],
    "resolution_question": "What is the recommended initial IV furosemide dose...",
    "filled_page": "entities/furosemide.md",
    "filled_section": "Dosing",
    "retrieval_verified": false,
    "verified_score": 0.51,
    "filled_at": "2026-05-14T11:00:00Z"
  }
]
```

Written by the gap resolver when it closes a gap. Read by CDS Step 2 before `search_sections()`.

### Invalidation

If a wiki page is modified or deleted, any resolved gap index entries pointing to that page must be re-verified. On page write, run `search_sections(resolution_question)` for each affected entry and update `retrieval_verified` and `verified_score`. Entries where the section no longer exists are removed from the index.

### What this is NOT

This is not caching LLM answers. It is caching the **retrieval path** (query → section). The grounding LLM always reads live wiki content. Stale answers are not possible; broken pointers (deleted/moved sections) are caught by invalidation.

### Frontend implications

The Retrieval Health tab gains a second meaning: it shows not just what is broken, but what has been **learned and is actively being used**. Add a summary line to the tab header:

```

```

On each verified card, add a badge:

```

    Similarity threshold: 0.92 · 3 queries routed via shortcut this week
```

On each unverified card, the shortcut is still active but flagged:


This distinguishes between "we're working around the problem" (shortcut active, unverified) and "the problem is solved" (shortcut active, verified).

## Implementation Order

1. **Backend schema** — extend `write_gap_files()` to store `searched_sections` and `query_embedding`; add embedding-based dedup check
2. **CDS trace** — pass all retrieved sections (path + section + score) through to gap metadata instead of just `top_score` + page paths
3. **Post-fill verification** — add retrieval smoke test to `gap_resolver.py` after each write; store `retrieval_verified`, `verified_score`, `retrieval_mismatch`; write entry to `resolved_gap_index.json`
4. **Two-tier retrieval** — in CDS Step 2, check `resolved_gap_index.json` before calling `search_sections()`; add page-write invalidation hook
5. **Graph API** — expose `retrieval_mismatch` and `unverified_fill` per node; add mismatch edge list
6. **Frontend: Why panel** — expand wiki grounding block to show all sections + scores
7. **Frontend: Gap Intelligence** — enrich Open Gaps and Persistent KGs cards; add Retrieval Health tab with shortcut activity counts
8. **Frontend: Graph** — mismatch edges, node rings, retrieval filter, tooltip, stacked bar chart
